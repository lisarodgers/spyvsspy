\# Spy vs Spy Computers



\# UX\_GUIDELINES.md



\## Purpose



The purpose of these guidelines is to create a user experience that is calm, trustworthy, educational, and professional.



Cybersecurity software often overwhelms users with technical jargon, flashing warnings, and unnecessary fear.



Spy vs Spy Computers takes a different approach.



Our software should help users understand what happened, why we believe it happened, and what they can do next.



Every interface should reduce anxiety and increase understanding.



\---



\# Design Philosophy



Our interface should feel like a professional investigation desk.



Not a hacker movie.



Not a toy.



Not a command line disguised as a website.



Think:



\* Modern

\* Premium

\* Minimal

\* Fast

\* Confident

\* Purposeful



\---



\# Brand Personality



Spy vs Spy Computers is:



\* Intelligent

\* Calm

\* Curious

\* Transparent

\* Professional

\* Educational

\* Evidence-driven



We are never:



\* Alarmist

\* Fear-based

\* Sensational

\* Condescending

\* Overly technical



\---



\# User Experience Principles



\## Show Evidence Before Conclusions



Never simply display:



> High Risk



Instead display:



Threat Score



Evidence



Explanation



Confidence



Recommendation



Every conclusion must answer:



\* What happened?

\* Why do we believe it?

\* How confident are we?

\* What should the user do?



\---



\## Explain Every Score



Threat Score



92



should always include:



Because we found:



\* Bitcoin wallet

\* Null Return Path

\* Failed SPF

\* Known campaign

\* Suspicious SMTP route



Users should never wonder how a score was calculated.



\---



\## Educate Without Interrupting



Every technical term should have:



\* Tooltip

\* Help icon

\* Expandable explanation



Example:



SPF



"What is SPF?"



Click



Simple explanation.



Advanced explanation.



Links to documentation.



\---



\## Progressive Disclosure



Beginners should not be overwhelmed.



Advanced users should not feel limited.



Default view:



Simple.



Advanced mode:



Everything.



\---



\## Trust Through Transparency



Never hide information.



If something was ignored:



Say so.



If confidence is low:



Say so.



If evidence is incomplete:



Say so.



Honesty builds trust.



\---



\# Visual Style



\## Theme



Dark Mode First.



Primary colors:



Black



White



Red



Accent colors should be subtle.



Red indicates importance, not decoration.



Avoid excessive neon effects.



Elegance over flash.



\---



\## Typography



Modern sans-serif fonts.



High contrast.



Comfortable spacing.



Readable at all sizes.



Avoid tiny text.



\---



\## Layout



Use cards.



Use whitespace.



Avoid clutter.



Group related information together.



Each screen should answer one primary question.



\---



\# Icons



Use icons only when they improve recognition.



Never rely on icons alone.



Always include labels.



\---



\# Navigation



Navigation should be shallow.



The user should reach any major feature in two or three clicks.



Primary sections:



\* Dashboard

\* Analyze

\* History

\* Intelligence

\* Reports

\* Settings



\---



\# Investigation Workflow



Every investigation should follow the same structure.



1\. Upload

2\. Validation

3\. Analysis

4\. Evidence

5\. Explanation

6\. Recommendation

7\. Save



The order should remain consistent across all investigation types.



\---



\# Reports



Reports should read like professional investigation summaries.



Avoid technical dumps.



Organize information into sections.



Overview



Evidence



Technical Details



Risk Assessment



Recommended Actions



References



\---



\# Colors



Green



Confirmed Safe



Yellow



Needs Review



Orange



Suspicious



Red



Confirmed Threat



Gray



Unknown



Never rely on color alone.



Include icons and text.



\---



\# Language



Prefer:



"We found..."



"We observed..."



"The evidence suggests..."



Avoid:



"You have been hacked."



"This is definitely malware."



Unless the evidence supports those conclusions.



\---



\# Recommendations



Recommendations should always be practical.



Good:



Delete the email.



Verify the sender independently.



Report the message.



Review SPF records.



Bad:



"Contact an expert."



Without explaining why.



\---



\# Performance



The interface should feel responsive.



Provide immediate feedback.



Show progress during long operations.



Never leave the user wondering whether something is happening.



\---



\# Accessibility



Support:



\* Keyboard navigation

\* Screen readers

\* High contrast

\* Scalable fonts

\* Color-blind friendly indicators

\* Clear focus states



Accessibility is part of good design.



\---



\# Mobile



The application should remain usable on tablets.



Desktop remains the primary experience.



Do not sacrifice desktop usability to optimize for mobile.



\---



\# Empty States



Every empty page should teach.



Instead of:



"No investigations."



Use:



"No investigations yet."



"Upload your first email to begin building your investigation history."



\---



\# Error Messages



Good errors explain:



What happened.



Why it happened.



How to fix it.



Never expose stack traces to users.



\---



\# Success Messages



Celebrate progress without becoming distracting.



Example:



Investigation Complete



12 indicators analyzed.



Threat Score: 91



Report ready.



\---



\# The Golden Rule



Every screen should answer three questions:



1\. What happened?

2\. Why do we believe that?

3\. What should the user do next?



If the interface cannot answer those questions clearly, it needs to be redesigned.



\---



\# User Promise



Spy vs Spy Computers exists to help people understand digital evidence.



We do not make decisions for users.



We present the facts.



We explain the reasoning.



We respect the user's ability to think critically.



\---



\# Mission



\*\*Spy vs Spy Computers doesn't tell people what to think.\*\*



\*\*It shows them the evidence and explains why.\*\*
